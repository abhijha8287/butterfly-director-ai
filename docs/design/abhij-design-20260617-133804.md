## Problem Statement

Butterfly Director AI turns a single story premise into a branching multiverse of fully generated short films — not just branching text or storyboards, but actual edited movies (video + voice + music) for every alternate timeline. The full system architecture already exists (`ARCHITECTURE.md`) along with eight detailed follow-on specs (backend foundation, multi-agent system, timeline engine, movie production pipeline, frontend, production infrastructure, performance/observability, hackathon polish). None of it is built yet — the repo contains only `ARCHITECTURE.md`. The goal of this session was not to add a ninth spec, but to figure out which slice of the existing eight is actually buildable before a hackathon deadline, and in what order.

## What Makes This Cool

Existing branching-narrative AI tools (ChronoWeave, Multiverse Explorer, Glimora — see Cross-Model Perspective below, sourced via web search) stop at text, images, or simple audio per branch. None of them assemble a fully edited, watchable **film** per universe. Butterfly Director AI's real differentiator isn't the branching mechanic (commoditized) — it's finishing the pipeline all the way to a real movie, for every fork, with consistent characters across universes. The chosen "whoa" moment for the demo is watching two generated movies from the *same* story diverge — same characters, different fates — side by side.

## Constraints

- Hackathon deadline: real time is finite; eight specs each described as standalone "production-ready, no placeholders" cannot all be built before judging.
- Real video generation (Wan/HappyHorse) is slow and costly per clip — this is the actual bottleneck, not the branching logic.
- Judges need to open a public URL themselves, without the builder present to drive a local demo.
- Hard constraint carried through every spec in this project: no mocked data, no placeholder implementations — every page must talk to a real backend.

## Premises

1. The core differentiator worth proving is "every branch becomes a real finished movie" — branching narrative alone is commoditized; full video assembly per branch is what nobody else does. **(confirmed)**
2. Build priority should weight toward the video/voice/music/Editor pipeline actually working end-to-end, and the demo should bound total branch count (e.g. one story → one decision → 3-4 branches) rather than letting Decision Detector cascade into dozens of branches — judging time and render cost both scale with branch count. **(confirmed)**
3. No code exists yet — the repo only contains `ARCHITECTURE.md`. All eight specs are still fully greenfield. **(confirmed)**
4. For hackathon judging, a reliably-running live demo matters far more than a landing page, pitch deck, or cloud deployment ceremony. **(confirmed, and refined: a public URL is required, so "reliably running" includes "reachable without the builder present" — see Distribution Plan)**

## Cross-Model Perspective

(Web search, not a second-model review — used in place of competitive research per Phase 2.75.) Searched generic category terms ("branching narrative AI video generator," "AI multiverse story generator interactive timeline graph tool"). Findings:
- **Layer 1 (already known):** Branching narrative tools (Twine, ink, visual novels) and AI video generators (Sora, Runway, Wan) have existed for years, as separate categories.
- **Layer 2 (current state):** 2026 tools like ChronoWeave and Multiverse Explorer do multiverse "what-if" branching with text+images; Glimora generates text+art+audio per branch. The field's design philosophy is converging on *constrained* branching (world state, character memory, authored rules) rather than unconstrained generation — this directly validates the Character Memory / Character Consistency Agent design already in `ARCHITECTURE.md` and the multi-agent spec.
- **Layer 3 (the gap):** Every tool found stops at text, image, or audio per branch. None assembles a fully edited short film per universe.
- **Eureka:** Everyone in this space treats a "branch" as a piece of content because video assembly per branch is assumed too expensive to do for every fork. Butterfly Director AI is built on the premise of actually paying that cost — which is the real wedge, not the branching mechanic itself.

## Approaches Considered

### Approach A: Minimal Viable, Simplest Infra
Same lean infrastructure as Approach C, but also trims agent scope (folding some agents together, skipping Critic/Music as separate agents initially). Fastest to ship, but the multi-agent sophistication — part of the pitch — becomes less visible.
- Effort: S-M. Risk: Low.
- Pros: Fastest path to a working end-to-end demo; least to debug under deadline pressure.
- Cons: Visibly less impressive agent pipeline; undersells the "multi-agent orchestration" half of the pitch.

### Approach B: Full Architecture Exactly As Specified
Build literally everything across all eight specs: six separate Celery worker pools, Qdrant wired in from day one, full Alibaba ECS/RDS/Container Registry/Log Service, Prometheus/Grafana/OpenTelemetry.
- Effort: XL. Risk: High.
- Pros: Matches the long-term architecture exactly; no rework needed if this becomes a real product later.
- Cons: Far more surface area than can realistically be completed and verified before a deadline; the extra infrastructure doesn't make the live demo more impressive, it just adds failure points that can break on judging day.

### Approach C: Lean Infra, Full Creative Pipeline (chosen)
Single Celery worker process consuming all queues (not six separate pools), defer Qdrant until Character Memory actually needs semantic retrieval, one Postgres + one Redis instance, deployed on a single reachable host with a public URL. But build all 9 real agents (Story Architect, Character Architect, Decision Detector, Timeline Generator, Character Consistency, Storyboard, Prompt Director, Video Generation, Voice, Music, Editor, Critic) and the full real video/voice/music/edit chain faithfully — no shortcuts inside the creative pipeline itself.
- Effort: M-L. Risk: Low-Medium.
- Pros: The part judges actually evaluate (does the multiverse/video pipeline genuinely work) gets full attention; infra that doesn't show up in a demo (six worker pools, Grafana dashboards) is deferred without anyone noticing; still 100% real, zero mocked responses anywhere.
- Cons: Needs careful validation that real video generation latency per branch fits within acceptable build/test/demo time; defers spec #6/#7's enterprise observability and multi-AZ cloud setup entirely for now (logged as Open Questions/follow-up, not silently dropped).

## Recommended Approach

**Approach C.** It is the only option consistent with everything confirmed this session: real (no-mock) full creative pipeline, a bounded but still "all branches get real video" demo, a publicly reachable URL, and the explicit premise that a working demo beats infrastructure polish.

## Open Questions

1. **Resolved during review:** branch-count bound is locked at `max_branches_per_decision = 4`, `max_decisions_per_story = 1` for the hackathon build (a single decision point, up to 4 branches). All later references to "3-4 branches" in this doc refer to this locked default, not an illustrative placeholder. Revisit only if the demo needs a second decision layer.
2. Real per-clip video generation latency from Wan/HappyHorse is unknown until actually tested against their APIs. Contingency: budget roughly 5-15 minutes per branch end-to-end (storyboard → video → voice → music → edit) as a planning estimate — confirm against real API latency in week 1 of implementation. If a branch is still rendering when a judge opens the demo, the Movie Player and Timeline Explorer must show real in-progress job status (per `ARCHITECTURE.md`'s job/progress model), never a fake "completed" state. The demo's sample project should be generated for real ahead of judging day specifically to avoid live-render risk; this counts as satisfying "no mocked responses" because the content is genuinely generated by the real pipeline, just not generated *during* the judging window. A user-initiated *new* story must still run the real pipeline live, even if slower.
3. Which single host/PaaS to deploy to is not yet decided — a single Alibaba ECS instance keeps the deploy within the existing cloud vendor (already used for OSS) without the full ECS/RDS/Registry/LogService ceremony from spec #6.
4. API key availability and rate limits/quotas for Qwen, Wan, and HappyHorse have not been confirmed. Contingency: confirm all three before writing any agent code — this is a hard go/no-go blocker for "no mocked responses." If any provider is unavailable, the fallback is substituting a different real provider with an equivalent capability (e.g. another video-gen API), never a mocked response, since mocking violates the core constraint carried through every spec in this project.
5. **Agent scope justification (added during review):** all 9 agents from the multi-agent spec are kept because each is load-bearing for the chosen demo moment specifically: Critic Agent catches continuity/quality errors before two branches are shown side by side — without it, a broken branch turns the "whoa" moment into a "wait, that's broken" moment; Music Agent shapes the emotional read of each branch, which is part of what makes two diverging endings feel genuinely different rather than cosmetically different. If the time budget (Open Question 2) doesn't hold, Critic and Music are the first two agents to simplify (rule-based checks / pre-selected mood tracks instead of full LLM-driven generation) before cutting anything from the core branching-to-video chain.
6. If real-time per-branch latency (Open Question 2) turns out too slow even for a pre-generated sample project within available build time, the fallback is reducing `max_branches_per_decision` from 4 to 2 — never switching to mocked output.

## Success Criteria

- A user can enter a story premise and receive a real LLM-generated story bible plus at least one real detected decision point (no canned/template output).
- That decision produces multiple real branches (universes) as actual `Branch`/`Movie` rows in the database and as live nodes in the Timeline Explorer graph.
- Every generated branch gets a fully rendered short movie via real Wan/HappyHorse video generation, real voice synthesis, real music generation, and real edit assembly — stored in real Alibaba OSS, not local mocks.
- The Timeline Explorer (React Flow) renders the graph live and supports the chosen "whoa" moment: side-by-side comparison of two universes generated from the same premise.
- The whole system is reachable by a public URL that judges can open without the builder driving it locally.
- Zero mocked API responses anywhere in the frontend — every page genuinely calls the FastAPI backend.

## Distribution Plan

Single-host deployment: one VM (Alibaba ECS instance, keeping it within the same cloud vendor already used for OSS) running the full `docker-compose` stack (FastAPI, Next.js, Postgres, Redis, single Celery worker + beat, Nginx as reverse proxy/TLS termination). No Container Registry, no managed RDS cluster, no auto-scaling — those are deferred per Approach C. A domain or the instance's IP plus a TLS cert (Let's Encrypt via Nginx) gives judges a stable public URL. CI/CD is out of scope for this slice; deploy is a manual `docker compose up -d` on the host until/unless time remains for spec #6's GitHub Actions pipeline.

## Next Steps

1. Finish the backend foundation (already-queued tasks: config/logging/middleware, DB models, repositories/schemas/services/routers, Celery integration, Alembic migration, Docker artifacts) using Approach C's lean infra — one worker process, Qdrant deferred.
2. Fold the Timeline Engine's `TimelineManager` / Decision / Branch / Butterfly-score logic directly into the backend's services and models rather than building it as a separate system — with branch-count bounds configured from day one (resolves Open Question 1).
3. Build the 9-agent LangGraph pipeline for real, wired to actual Qwen/Wan/HappyHorse calls: Story Architect → Character Architect → Decision Detector → Timeline Generator → Character Consistency → Storyboard → Prompt Director → Video Generation → Voice → Music → Editor → Critic.
4. Build the four demo-critical frontend screens only: Dashboard, Create Project Wizard, Timeline Explorer (with side-by-side universe comparison), Movie Player. Defer Character Manager, Asset Library, Agent Monitor, Settings, landing page unless time remains.
5. Resolve Open Questions 2-4 (video latency, host choice, API key availability) early — they gate whether Approach C is actually achievable as scoped.
6. Deploy single-host with a public URL; run a full end-to-end smoke test (premise → branches → movies → comparison) before judging day.
7. Spec #8 (hackathon polish) floor vs. stretch: the floor (must-have for submission) is README, one or two architecture diagrams, and FastAPI's auto-generated OpenAPI docs — nothing else is required to submit. Everything else in spec #8 (pitch deck, landing page, demo script, presentation assets) is stretch, attempted only after Next Steps 1-6 are solid and verified working end-to-end.
